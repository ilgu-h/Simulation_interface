# Comm Group Configuration & Topology Auto-Derive

**Date:** 2026-04-16
**Status:** Design approved

## Problem

When running ASTRA-sim with workloads that use multiple parallelism dimensions (DP > 1, TP > 1, etc.), the simulation crashes:

```
[workload] [critical] For rank 0 ET node 2426, communicator group 1 not found
```

**Root cause:** Two issues compound:

1. **Critical bug:** `astra_runner.py` does not pass `--comm-group-configuration` to ASTRA-sim. The `workload.json` file (generated by STG with comm group definitions) is never loaded. ASTRA-sim defaults to `"empty"`, clearing all comm groups.
2. **UX gap:** The System page auto-fills a flat 1D network (`npus_count=[N]`) regardless of the workload's parallelism structure. Users must manually configure multi-dimensional topology to match their parallelism dims.

## Solution

### Part 1: Pass comm-group-configuration to ASTRA-sim (bug fix)

**File:** `backend/app/orchestrator/astra_runner.py`

Add `comm_group_config` to `AstraInvocation` and include `--comm-group-configuration` in the CLI args.

The comm group file path is derived from the workload prefix:
- Workload prefix: `runs/<id>/traces/workload`
- Comm group file: `runs/<id>/traces/workload.json`

```python
@dataclass(frozen=True)
class AstraInvocation:
    binary: Path
    workload_prefix: Path
    comm_group_config: Path      # NEW
    system_config: Path
    network_config: Path
    memory_config: Path
    logging_folder: Path

    def cli(self) -> list[str]:
        return [
            str(self.binary),
            f"--workload-configuration={self.workload_prefix}",
            f"--comm-group-configuration={self.comm_group_config}",  # NEW
            f"--system-configuration={self.system_config}",
            f"--network-configuration={self.network_config}",
            f"--remote-memory-configuration={self.memory_config}",
            f"--logging-folder={self.logging_folder}",
        ]
```

In `build_invocation()`:
```python
comm_group_path = workload_prefix.parent / f"{workload_prefix.name}.json"
# If comm group file doesn't exist, use "empty" (ASTRA-sim default)
comm_group_config = comm_group_path if comm_group_path.exists() else "empty"
```

### Part 2: Auto-derive network topology from parallelism (UX improvement)

**File:** `frontend/app/system/page.tsx`

When navigating from the Workload page, URL params already carry `dp`, `tp`, `sp`, `pp`, `ep`. Replace the current flat auto-fill with multi-dimensional derivation.

**Mapping rule:**
```
For each dim in [DP, TP, SP, PP, EP(if moe)]:
  If value > 1 -> add as a network dimension

All dims == 1 -> fallback: topology=[Ring], npus_count=[total_npus]
```

**Examples:**

| Parallelism | Active dims | Network topology | npus_count |
|---|---|---|---|
| dp=1, tp=4, sp=1, pp=1 | TP=4 | [Ring] | [4] |
| dp=2, tp=4, sp=1, pp=1 | DP=2, TP=4 | [Ring, Ring] | [2, 4] |
| dp=2, tp=2, sp=1, pp=2 | DP=2, TP=2, PP=2 | [Ring, Ring, Ring] | [2, 2, 2] |
| dp=1, tp=1, sp=1, pp=1 | (none) | [Ring] | [total_npus] |

**UI changes:**
- Each network dimension row shows a label: "Dim 0 (DP)", "Dim 1 (TP)", etc.
- Default topology type: `Ring` for all dimensions
- Default bandwidth: 50.0 GB/s, default latency: 500.0 ns (current defaults)
- User can freely edit all fields (topology type, bandwidth, latency, even npus_count)
- Show an info banner if npus_count values are edited away from the derived values

### Part 3: Validation enhancement

**File:** `backend/app/api/runs.py`

In `POST /runs/validate`, add checks:

1. If workload has a `workload.json` file, verify it's non-empty when any spatial parallelism dim > 1
2. Warn (not block) if `len(topology)` < number of active parallelism dims

## Non-goals

- Modifying STG to accept topology parameters (STG's comm group generation is already correct)
- Changing ASTRA-sim's CommunicatorGroup logic (it handles sub-cluster groups via internal Ring topologies)
- Embedding the comm group structure in the network config YAML (ASTRA-sim uses a separate file)
